# Results Guide

Generated outputs live under `results/`. The directory is ignored by git and
should not be committed.

## Common Files

- `trajectory.csv`: per-frame time series.
- `summary.json`: per-run metrics and stop reason.
- `run_config.yaml`: configuration used by a single run.
- `all_runs_summary.csv` or `summary.csv`: batch-level run table.
- `aggregate_summary.json` or `aggregate_by_condition.csv`: grouped metrics.
- `*.png`: plots generated by the run or batch script.

## Important Files by Step

Step 1 live runs write `trajectory.csv`, `summary.json`, `run_config.yaml`,
`distance_plot.png`, and `trajectory_plot.png`.

Step 1 batch runs write `all_runs_summary.csv`, `aggregate_summary.json`,
distance/error plots, duration plots, and one folder per target/repetition.

Step 2A live and batch runs add current metadata, lateral drift plots, speed
plots, and current-conditioned aggregate summaries.

Step 2B live runs add velocity tracking and command plots. Step 2B comparison
runs write one subfolder per mode plus `all_runs_summary.csv`,
`aggregate_summary.json`, and comparison plots.

Step 2C live runs add PI command metadata, saturation metrics, and the same
distance / velocity / command plots. Step 2C comparison runs add target-reached
rate and saturation summary plots.

Step 2C `saturation_fraction` and `sway_saturation_fraction` are saturation
relative to software command limits, not verified physical T200 saturation.

Step 3 live runs write `trajectory.csv`, `summary.json`, and
`run_config.yaml`. Step 3 batches write `summary.csv`,
`aggregate_by_condition.csv`, `metadata.json`, `metadata.yaml`, `logs.txt`, and
altitude / drift plots.

Step 3B OpenWater runs write `all_runs_summary.csv`,
`aggregate_summary.json`, `metadata.json`, one folder per named transect, and
`openwater_hole_profile.png` for each transect.

## Core Metrics

- `target_reached`: true only when the experiment stopped because the target was reached.
- `stop_reason`: terminal condition such as `target_reached`, `timeout`, `collision`, `unsafe_altitude`, `invalid_ping_failure`, or `exception`.
- `timeout`: true when the target was not reached before `max_duration`.
- `lateral_drift`: side displacement relative to the starting body-frame right axis.
- `final_position_error`: final Euclidean distance from the target point, used mainly in Step 2 comparisons.
- `duration` or `runtime_seconds`: simulated runtime until stop.

Timeout runs should remain in summaries. They are failed target-reaching runs,
not missing data.

## Step 2C Reduction Validity

Step 2C comparison reports raw drift/error reductions versus no compensation.
Reduction percentages are valid only when both compared modes reached the target
for that target/current group. If either mode timed out, keep the raw values but
treat the validity flag as false.

Step 2B 60 s historical results should not be directly compared against Step 2C
120 s results. Modern Step 2B and Step 2C comparisons should both use
`--max-duration 120`.

## Step 3 Altitude Metrics

The main Step 3 altitude-hold metrics are:

- `rmse_altitude_error`
- `max_abs_altitude_error`
- `time_inside_altitude_band_percent`
- `time_below_safe_altitude_percent`
- `vertical_saturation_percent`
- `target_reached`
- `timeout`
- `collision`
- `unsafe_altitude`
- `invalid_ping_failure`

Success means the vehicle advances by the requested distance while keeping
PingAltimeter altitude close to the desired altitude.

## Step 3B Terrain Metrics and Plots

Step 3B adds:

- `estimated_seabed_z_from_pose_ping` in `trajectory.csv`
- `estimated_seabed_z_min`
- `estimated_seabed_z_max`
- `estimated_seabed_z_range`
- `estimated_seabed_z_std`

The OpenWater profile plot shows Ping altitude, desired altitude, altitude
error, estimated seabed z, Pose z, and vertical command. Pose is used only for
offline terrain reconstruction and never for control.
